## Dominant Throughput Ceiling

`live HTTP`

The ring-only and adapter-only lanes are orders of magnitude faster than the external durable command path, and the shard-count comparison shows runtime parallelism now scales materially, so the remaining measured ceiling is in the live HTTP end-to-end path rather than disruptor handoff.

## Before/After Interpretation

- Before Plan 13.1-01, the research and prior measurements showed `shard_count=1` and `shard_count=8` behaving nearly the same because routed commands still converged on one runtime loop.
- After Plans 13.1-01 and 13.1-02, the same unique baseline workload improved from `106.62 req/s` at `shard_count=1` to `442.01 req/s` at `shard_count=8`, with `p50_micros` dropping from `74175` to `13431`.
- That shard-count gain is the strongest evidence that the runtime bottleneck identified in research was addressed.
- The remaining ceiling is not the raw ring: ring-only publish/poll stays around `78 ns`, adapter-only decode/admission stays around `2.07 µs`, while the live HTTP lane remains in the tens of milliseconds because it still includes durable append, real HTTP request/response work, and bounded command admission.

## Remaining Limits

- `storage-only.txt` lacks a PostgreSQL-only measurement in this environment because `DATABASE_URL` was not set for Criterion storage benches.
- `metrics_scrape_successes` remained zero in all baseline live HTTP outputs, so `append_latency_p95_micros`, `ring_wait_p95_micros`, and `shard_depth_max` cannot yet separate PostgreSQL append cost from metrics-collection gaps.
- `live-http-single-hot-key.json` remains a failure-heavy workload because repeated `PlaceOrder` requests revisit the same order stream; it is useful for contention diagnostics, not for success-only throughput claims.
- A future follow-up should rerun the baseline comparison with successful metrics scrapes and a valid `DATABASE_URL` for `storage_only` if the team wants to split the current live HTTP ceiling more precisely between PostgreSQL append and HTTP overhead.

## Next Phase Readiness

- Phase 13.1 now closes with one repeatable evidence workflow, documented interpretation rules, and archived shard-count proof that the runtime parallelism fix changed throughput materially.
- The remaining open question is not whether the ring is fast enough; it is how much of the live HTTP ceiling belongs to durable append versus other external-process command-path costs once storage-only and scrape-derived metrics are fully available.
