# Phase 13.1 - Security

Per-phase security verification for the disruptor throughput bottleneck investigation and runtime stress optimization. This audit verifies only the threats declared in the six Phase 13.1 `<threat_model>` blocks and the executor `## Threat Flags` sections.

## Trust Boundaries

| Boundary | Description | Primary Data Crossing |
|----------|-------------|-----------------------|
| adapter ingress -> runtime dispatcher | Accepted user commands cross into bounded internal queues and must receive a determinate outcome during shutdown. | Command envelopes, metadata, reply channels |
| runtime dispatcher -> shard workers | Work ownership changes from dispatcher to shard-owned workers. | Routed commands, shard IDs, per-shard queue depth |
| shard worker -> durable event store | In-flight commands must remain commit-gated during shutdown. | Domain events, expected revisions, append replies |
| Prometheus metrics -> stress report JSON | Observability scrape text becomes archived evidence. | Queue depth, latency, lag, scrape counters |
| CLI/report JSON -> operators and archive summaries | Machine-readable reports become source evidence for throughput conclusions. | Observed metrics, estimates, workload labels, host metadata |
| workload shape config -> request generator | Configured key distribution controls what the live workload actually proves. | Workload shape, hot-set size, generated order streams |
| benchmark harness -> storage database | Storage-only lane must measure the real durable append path. | Event-store append/read operations and benchmark timings |
| comparison script -> archived evidence files | Scripted outputs determine which artifacts future docs and audits cite. | Fixed lane filenames, JSON reports, Criterion text output |
| archived evidence -> docs and summary | Final written claims must not exceed regenerated measurement quality. | Ceiling classification, scrape-gated interpretation rules |
