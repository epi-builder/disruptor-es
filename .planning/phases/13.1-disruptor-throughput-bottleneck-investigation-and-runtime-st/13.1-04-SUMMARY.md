# Phase 13.1 Plan 4: Runtime Shutdown Drain Summary

**Deterministic runtime shutdown that drains accepted ingress with `RuntimeError::Unavailable` while preserving commit-gated shard completion**

## Accomplishments

- Added two runtime regressions covering shutdown with an accepted-but-undispatched command and shutdown with already-dispatched work blocked on the append gate.
- Updated `CommandEngine::run` to enter an explicit shutdown mode, close ingress, drain buffered envelopes with `RuntimeError::Unavailable`, and only then tear down shard senders.
- Verified the official `app serve` path still relies on `engine.run(shutdown)` in `crates/app/src/serve.rs`; no custom shutdown drain logic was added there.

## Task Commits

1. **Task 1: Add deterministic shutdown regressions for accepted and dispatched commands** - `56e238d` (`test`)
2. **Task 2: Drain undispatched ingress and preserve commit-gated worker completion during shutdown** - `afa6c47` (`feat`)

## Files Created/Modified

- `crates/es-runtime/tests/runtime_flow.rs` - Added deterministic shutdown regressions using `Notify` and append gates.
- `crates/es-runtime/src/engine.rs` - Added shutdown state, ingress close-and-drain handling, and prioritized shutdown selection.
- `crates/app/src/serve.rs` - Verified existing `engine.run(shutdown)` wiring remained the official lifecycle path; no code change required.

## Verification

- `cargo test -p es-runtime accepted_but_undispatched_commands_receive_unavailable_on_shutdown -- --nocapture`
- `cargo test -p es-runtime dispatched_commands_finish_before_engine_shutdown_returns -- --nocapture`
- `cargo test -p es-runtime runtime_flow -- --nocapture`

## Decisions Made

- Closed the runtime receiver as soon as shutdown starts so post-shutdown `try_submit` calls resolve through the existing `RuntimeError::Unavailable` path instead of racing new accepted work into the drain.
- Used a biased `tokio::select!` with the shutdown branch first so the drain-or-fail contract is deterministic even when shutdown and ingress readiness happen in the same poll.

None - plan executed exactly as written.
