# Phase 13.1: Code Review Report

**Reviewed:** 2026-04-27T00:00:00Z
**Depth:** standard
**Files Reviewed:** 11
**Status:** issues_found

## Summary

Reviewed the Phase 13.1 stress/runtime additions across the live HTTP harness, CLI, service composition, runtime engine, docs, and comparison script.

The main risks are benchmark correctness and shutdown behavior. Two of the new live-HTTP metrics paths can produce misleading evidence for the archived throughput analysis, and the updated engine shutdown path can strand already-accepted commands because it stops dispatching immediately on shutdown notification. Targeted existing tests passed, but they do not cover these edge cases.

## Warnings

### WR-01: Hot-set and single-hot-key workloads reuse order stream IDs and turn into domain failures

**File:** `crates/app/src/http_stress.rs:344-359`
**Issue:** The new workload-shape generator reuses `order_id` whenever `HttpWorkloadShape::HotSet` or `SingleHotKey` is selected. In this codebase, order routing and stream identity are derived from `order_id`, and `PlaceOrder` uses `ExpectedRevision::NoStream`. After the first successful request for a given key, later requests for that same key are no longer measuring hot-shard pressure; they become business-level `AlreadyPlaced`/bad-request failures. That makes `hot-key` and `hot-set` runs poor throughput comparators and can heavily skew `commands_failed`, latency, and bottleneck conclusions.
**Fix:**
```rust
// Keep the routed hot key stable without reusing a create-once order stream.
// One option is to benchmark a command whose partition key can stay hot while
// the stream id remains unique per request.

// Pseudocode:
match workload_shape {
    HttpWorkloadShape::Unique => make_unique_order(index),
    HttpWorkloadShape::HotSet(size) => make_unique_stream_with_hot_partition(index, size),
    HttpWorkloadShape::SingleHotKey => make_unique_stream_with_fixed_partition(index),
}
```
Add an integration test that runs at least two hot-key requests and asserts they are both valid benchmark requests rather than predictable domain failures.

### WR-02: `ingress_depth_max` is artificially floored to concurrency, so reports can claim queue pressure that never happened

**File:** `crates/app/src/http_stress.rs:461-464`
**Issue:** The report forces `ingress_depth_max` to be at least `min(concurrency, commands_submitted)` even when the scraped metric never observed that depth. If metrics scraping fails or the queue stays shallow, the report still emits a non-trivial ingress depth. That corrupts one of the archive-facing fields used to explain the throughput ceiling and can misclassify bounded admission as a bottleneck.
**Fix:**
```rust
ingress_depth_max: measured.metrics.ingress_depth_max,
```
If you need a fallback, emit `0` plus the scrape failure counters, or add a separate field such as `ingress_depth_estimate` instead of mutating the observed metric.

### WR-03: Engine shutdown stops dispatching immediately and can drop accepted ingress commands

**File:** `crates/es-runtime/src/engine.rs:140-156`
**Issue:** `CommandEngine::run` breaks out of the dispatcher loop as soon as `shutdown.notified()` resolves, then clears shard senders and joins workers. Any commands already accepted into `self.receiver` but not yet forwarded to a shard are abandoned at that point. Because the accepted command still owns a live reply channel, callers can be left waiting until the runtime is torn down rather than receiving a deterministic reply. Phase 13.1 added the long-lived `app serve` path and new shutdown wiring, but there is still no regression test covering `run(..., shutdown)` with in-flight accepted commands.
**Fix:**
```rust
pub async fn run(mut self, shutdown: Arc<Notify>) -> RuntimeResult<()> {
    let mut shutting_down = false;

    loop {
        tokio::select! {
            _ = shutdown.notified(), if !shutting_down => {
                shutting_down = true;
            }
            routed = self.receiver.recv() => {
                let Some(routed) = routed else { break; };
                self.dispatch_routed(routed).await?;
                if shutting_down && self.receiver.is_empty() {
                    break;
                }
            }
        }
    }

    self.shard_senders.clear();
    // join workers...
}
```
Add a runtime integration test that submits commands, triggers shutdown before dispatch completes, and asserts every accepted command either finishes successfully or receives an explicit runtime error.
