<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| storage API -> PostgreSQL transaction | Validated but still untrusted append data enters durable tables. |
| duplicate command -> command_dedup | A replayed command attempts to reuse prior command identity. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-11 | Tampering | append SQL | mitigate | Use SQLx bind parameters only; acceptance criteria reject `format!` and `push_str` SQL construction. |
| T-02-12 | Information Disclosure | append/dedupe predicates | mitigate | Every stream, event, and dedupe query must bind `tenant_id`; dedupe tests prove cross-tenant idempotency isolation. |
| T-02-13 | Tampering / Repudiation | duplicate command replay | mitigate | Persist and return `command_dedup` committed result keyed by `(tenant_id, idempotency_key)` without appending duplicate events. |
| T-02-14 | Spoofing | metadata persistence | mitigate | Persist `command_id`, `correlation_id`, `causation_id`, and `tenant_id` from `CommandMetadata` into every event row. |
| T-02-15 | Denial of Service | transaction conflicts | mitigate | Reject empty appends before transaction and map OCC mismatch to `StoreError::StreamConflict` without inserting events. |
</threat_model>
