<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| storage caller -> PostgreSQL | Untrusted event payloads, metadata, tenant IDs, idempotency keys, and stream IDs cross into durable SQL storage. |
| test process -> Docker PostgreSQL | Tests create a real database dependency outside the Rust process. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | migration tenant keys | mitigate | Put `tenant_id` in every primary key, unique key, and read-supporting index where tenant data is stored. |
| T-02-02 | Tampering | SQL schema constraints | mitigate | Use PostgreSQL `CHECK`, `UNIQUE`, foreign key, and identity-column constraints to reject invalid revisions, empty event types, duplicate event IDs, and duplicate stream revisions. |
| T-02-03 | Tampering / Repudiation | command_dedup | mitigate | Use `PRIMARY KEY (tenant_id, idempotency_key)` and stored committed-result columns for duplicate command replay. |
| T-02-04 | Spoofing | event IDs | mitigate | Do not use DB UUID defaults; require Rust-generated UUIDs from storage API inputs per D-14 so tests and callers can control IDs. |
| T-02-05 | Denial of Service | event payloads | mitigate | Add JSONB payload columns now; later API plan must enforce `MAX_JSON_PAYLOAD_BYTES` before insert. This plan blocks unbounded schema-side text concatenation by avoiding dynamic SQL. |
</threat_model>
