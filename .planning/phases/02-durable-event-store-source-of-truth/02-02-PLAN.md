<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller -> storage API | Caller-provided payloads, metadata, tenant IDs, stream IDs, event types, and idempotency keys enter storage validation. |
| storage API -> database | Validated DTOs later become SQL bind parameters. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06 | Denial of Service | NewEvent payload | mitigate | Enforce `MAX_JSON_PAYLOAD_BYTES: usize = 1_048_576` in `NewEvent::new` before SQL persistence. |
| T-02-07 | Spoofing | tenant metadata | mitigate | Use `CommandMetadata.tenant_id` as the source of tenant truth on append DTOs; do not accept a separate append-level tenant string. |
| T-02-08 | Tampering | empty/no-op appends | mitigate | Return `StoreError::EmptyAppend`; Phase 02 append success must mean at least one durable event. |
| T-02-09 | SQL Injection | future SQL boundary | mitigate | API models carry typed values only; later SQL plan must use SQLx binds and must not concatenate values into SQL strings. |
| T-02-10 | Information Disclosure | read methods | mitigate | Public read signatures require `TenantId` for stream, global, snapshot, and rehydration reads. |
</threat_model>
