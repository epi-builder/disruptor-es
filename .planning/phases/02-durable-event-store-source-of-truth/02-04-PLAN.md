<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| storage caller -> read API | Caller-provided tenant IDs, stream IDs, cursors, and limits control durable reads. |
| snapshot writer -> snapshots table | Caller-provided snapshot payload and metadata enter durable storage. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-16 | Information Disclosure | read_global/read_stream | mitigate | Every read predicate includes `tenant_id`; tests prove tenant-b events do not appear in tenant-a global reads. |
| T-02-17 | Tampering | snapshots | mitigate | Snapshot writes are keyed by `(tenant_id, stream_id, stream_revision)` and use SQLx bind parameters with JSONB columns. |
| T-02-18 | Denial of Service | read limits | mitigate | Read methods accept explicit `limit: i64`; implementation must reject negative limits and use SQL `LIMIT $n`. |
| T-02-19 | Repudiation | projector/outbox cursor | mitigate | Global reads use durable `events.global_position`, not timestamps or disruptor sequences. |
| T-02-20 | Spoofing | metadata | mitigate | Snapshot and event rows preserve tenant-scoped stream identity; no cross-tenant snapshot lookup is allowed. |
</threat_model>
