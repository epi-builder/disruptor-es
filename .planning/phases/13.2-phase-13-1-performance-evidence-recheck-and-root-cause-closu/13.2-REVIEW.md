## Summary

Reviewed the Rust stress runner, CLI wiring, comparison script, supporting docs, and regenerated layer-comparison artifacts. The runtime/reporting code is broadly consistent and the existing `app` HTTP-stress tests passed, but the baseline comparison script is still generating two differently named artifacts from the same live-HTTP lane configuration, and the docs treat those files as distinct evidence.

## Warnings

### WR-01: Baseline comparison duplicates the 8-shard unique lane under two filenames

**File:** `scripts/compare-stress-layers.sh:64-71`
**Issue:** In baseline mode, `live-http-unique.json` and `live-http-shard-8.json` are both generated with `--profile baseline --workload-shape unique --warmup-seconds 5 --measure-seconds 30 --concurrency 8 --shard-count 8 --ingress-capacity 256 --ring-size 256`. The archived outputs confirm the duplication: both JSON files report the same lane metadata (`profile_name="baseline"`, `workload_shape="unique"`, `shard_count=8`, `ingress_capacity=256`, `ring_size=256`, `concurrency=8`). That means the Phase 13.2 docs in `docs/stress-results.md:123-125` and `docs/template-guide.md:167` overstate the breadth of the baseline evidence by treating three baseline success-throughput artifacts as three distinct comparison lanes when only two are distinct (`shard_count=1` and `shard_count=8`).
**Fix:**
```bash
# Make the files represent distinct lanes, or stop citing them as distinct evidence.
# For example, keep only one 8-shard unique artifact:
run_json_lane "$OUTPUT_DIR/live-http-shard-8.json" \
  cargo run -q -p app -- http-stress --profile baseline --workload-shape unique \
    --warmup-seconds 5 --measure-seconds 30 --concurrency 8 \
    --shard-count 8 --ingress-capacity 256 --ring-size 256

# Then either remove live-http-unique.json from the baseline script/docs,
# or change it to a genuinely different lane and update the documentation
# so the archive cites only distinct configurations.
```
