# Phase 13.2: Code Review Report

**Reviewed:** 2026-04-27T14:48:41Z
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

Reviewed the Phase 13.2 documentation and archived layer-comparison artifacts for evidence correctness, internal consistency, and traceability. The main issues are not code defects; they are evidence defects that overstate what the archived JSON can prove and make the shard/root-cause story hard to verify from the artifacts alone.

## Warnings

### WR-01: Storage-append ceiling claim is not backed by the cited live HTTP metrics

**File:** `docs/stress-results.md:123`
**Issue:** `docs/stress-results.md` and `docs/template-guide.md` both upgrade the archive label to `storage append / durable command path`, but every cited baseline live HTTP artifact records `append_latency_p95_micros: 0` (`target/phase-13.1/layer-comparison/live-http-unique.json:1`, `target/phase-13.1/layer-comparison/live-http-shard-1.json:1`, `target/phase-13.1/layer-comparison/live-http-shard-8.json:1`). That means the archive-safe files do not actually contain measured append-latency evidence for the bottleneck they claim is dominant. As written, the docs present an inference as if it were directly measured evidence.
**Fix:**
```md
The current Phase 13.2 label is `durable command path (append metric unavailable in live HTTP artifacts)`.

Only relabel this as `storage append` after regenerated live HTTP artifacts report non-zero
`append_latency_p95_micros` values or another explicit storage-bound metric.
```

### WR-02: The shard-count comparison is not self-describing in the archived JSON

**File:** `docs/template-guide.md:140-166`
**Issue:** The docs rely on `live-http-shard-1.json` versus `live-http-shard-8.json` to claim that "the one-shard lane slows materially before the eight-shard lanes flatten" (`docs/stress-results.md:123`), but neither JSON artifact includes `shard_count` or any equivalent lane metadata. Both files report the same `workload_shape: "unique"` schema-only view (`target/phase-13.1/layer-comparison/live-http-shard-1.json:1`, `target/phase-13.1/layer-comparison/live-http-shard-8.json:1`). A downstream reviewer cannot verify from artifact contents alone that these files differ by shard configuration rather than filename convention.
**Fix:**
```json
{
  "profile_name": "baseline",
  "workload_shape": "unique",
  "shard_count": 1,
  "ring_size": 256,
  "ingress_capacity": 256
}
```

### WR-03: The hot-key diagnostic artifact records total failure but no diagnosable cause

**File:** `target/phase-13.1/layer-comparison/live-http-single-hot-key-diagnostic.json:1`
**Issue:** The diagnostic lane is documented as evidence that can surface ordering or hot-key contention behavior (`docs/stress-results.md:50`, `docs/template-guide.md:162`), but the archived artifact only reports aggregate counters: `commands_failed: 13639`, `commands_succeeded: 0`, and `throughput_per_second: 0.0`. There is no failure taxonomy, HTTP status breakdown, or sample error payload. That makes the file insufficient for "root-cause closure" because it cannot distinguish OCC conflicts, domain validation failures, transport errors, overload, or an actual runtime bug.
**Fix:**
```json
{
  "commands_failed": 13639,
  "failure_kind_counts": {
    "conflict": 13639
  },
  "sample_failures": [
    {
      "kind": "conflict",
      "message": "expected revision mismatch"
    }
  ]
}
```
