<objective>
Create the durable replay substrate for Phase 08.

Purpose: PostgreSQL must be able to return the original committed command reply after a runtime restart or shard-cache miss without re-running `A::decide`.
Output: Store-owned replay DTOs, append-time replay payload persistence, public tenant/idempotency lookup, and durable integration tests.

Durable replay strategy: persist a versioned typed reply payload inside the existing `command_dedup.response_payload` JSONB value by storing `CommandReplayRecord { append, reply }` for new command appends. This is the simplest strategy that preserves API reply shape after restart without adding a new table or deriving replies from domain events. The store keeps backward-compatible append duplicate extraction from older `CommittedAppend` payloads, while typed replay lookup returns `None` for legacy append-only rows and returns `StoreError::DedupeResultDecode` for corrupted replay wrappers.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Runtime -> PostgreSQL command_dedup | Runtime-provided typed reply payload becomes durable replay data. |
| PostgreSQL command_dedup -> Runtime | Durable JSONB is decoded into typed replay records used to answer duplicate commands. |
| Tenant A -> shared store -> Tenant B | Tenant-scoped idempotency keys coexist in one durable table. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Tampering | `command_dedup.response_payload` | mitigate | Persist `CommandReplayRecord { append, reply }` only after append commit inside `insert_dedupe_result`; no replay row is written before durable append succeeds. |
| T-08-02 | Tampering / Information Disclosure | `lookup_command_replay` | mitigate | Every lookup binds both `tenant_id` and `idempotency_key`; `lookup_command_replay_is_tenant_scoped` proves same idempotency key under another tenant returns `None`. |
| T-08-03 | Tampering / Denial of Service | replay JSON decode | mitigate | Decode typed replay records with serde and return `StoreError::DedupeResultDecode` for invalid JSON shapes; legacy `CommittedAppend` rows remain append-only and do not produce typed replies. |
| T-08-04 | Tampering | outbox/process-manager duplicate effects | transfer | Storage replay support prevents duplicate append/outbox creation for runtime consumers; process-manager-specific retry behavior is verified in Plan 08-03. |
</threat_model>
