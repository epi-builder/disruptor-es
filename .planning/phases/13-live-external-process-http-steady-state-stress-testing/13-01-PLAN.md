    3. measured traffic for `measurement_seconds`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI/config input -> load generator | Untrusted local input could create unsafe or unbounded stress runs. |
| stress harness -> target service | The harness must load only its own spawned localhost service, not an arbitrary target URL. |
| Prometheus scrape/resource sampling -> report | Sampled metrics and metadata must reflect measured-window behavior without secret leakage. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Denial of Service | `HttpStressConfig` parsing and validation | mitigate | Add exact numeric bounds for warmup, measurement, concurrency, ingress capacity, shard count, command count, and ring size; reject invalid configs before any process starts. |
| T-13-02 | Denial of Service | external-process target selection | mitigate | Keep the runner deriving `listen_addr` and `prometheus_addr` only from `ExternalProcessHarness::spawn`; do not accept arbitrary target URLs or remote hosts. |
| T-13-03 | Information Disclosure | measured-window report metadata | mitigate | Record bounded metadata fields (`host_os`, `host_arch`, `cpu_brand`, `cpu_usage_samples`) and never serialize `DATABASE_URL`, env maps, or secret values. |
| T-13-04 | Denial of Service | metrics surface | mitigate | Reuse the bounded Phase 7 metric names and labels from `observability.rs`; do not add request-, tenant-, or command-identity labels during stress sampling. |
</threat_model>
