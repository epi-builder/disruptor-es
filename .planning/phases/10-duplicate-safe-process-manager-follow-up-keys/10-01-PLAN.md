<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| committed event store -> app process manager | `ProcessEvent.payload` is decoded into `OrderEvent`; malformed payloads must remain typed decode failures. |
| app process manager -> runtime command gateway | Follow-up command identity crosses into runtime/store replay as an opaque `idempotency_key`. |
| runtime -> event store replay lookup | Runtime/store replay uses `(tenant_id, idempotency_key)` and must not be bypassed by app-local dedupe. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | reserve follow-up key construction | mitigate | Task 1 changes reserve keys to `pm:{manager}:{source_event_id}:reserve:{line_index}:{product_id}`; Task 2 asserts duplicate same-product lines emit `reserve:0` and `reserve:1` keys. |
| T-10-02 | Tampering | release compensation key construction | mitigate | Task 1 stores `(line_index, product_id, quantity)` for successful reserves and uses `pm:{manager}:{source_event_id}:release:{line_index}:{product_id}`; Task 2 asserts duplicate-line failure releases `release:0:{product_id}`. |
| T-10-03 | Repudiation / Tampering | process-manager retry replay | mitigate | Task 3 keeps follow-ups through `CommandGateway` and real `CommandEngine`, with replay lookup keyed by exact idempotency key and append counts proving retries do not append fresh commands. |
| T-10-04 | Information Disclosure / Tampering | cross-tenant replay scope | accept | Phase 8/9 runtime/store replay remains tenant-scoped by `(tenant_id, idempotency_key)`; Phase 10 does not change tenant propagation and keeps `follow_up_metadata(event).tenant_id = event.tenant_id.clone()`. |
</threat_model>
