  <behavior>
    - `AppendRequest::new(...)` remains source-compatible and sets `outbox_messages` to `Vec::new()`.
    - `AppendRequest::new_with_outbox(...)` accepts `Vec<NewOutboxMessage>` and rejects any `NewOutboxMessage.source.event_id` that is not present in `events`.
    - `sql::append` inserts matching outbox rows after `insert_event` returns the source global position and before `insert_dedupe_result`.
    - Duplicate command replay returns `AppendOutcome::Duplicate` from `command_dedup` and does not insert new outbox rows.
  </behavior>
  <action>
    Modify `crates/es-store-postgres/src/models.rs` to import `es_outbox::NewOutboxMessage`, add `pub outbox_messages: Vec<NewOutboxMessage>` to `AppendRequest`, keep `AppendRequest::new(...)` with the current signature by delegating to a new `AppendRequest::new_with_outbox(...)`, and add validation that every outbox message source event ID matches one of the `events[*].event_id`. If validation fails, return a new `StoreError::InvalidOutboxSourceEvent { source_event_id: uuid::Uuid }`.

    Modify `crates/es-store-postgres/src/sql.rs` so `append` builds a map from each inserted `event_id` to `global_position` while inserting events. After each event insert or after the event loop, insert all `request.outbox_messages` using a private `insert_outbox_message(tx, request, message, source_global_position)` helper before `insert_dedupe_result`. The insert SQL must write `outbox_id`, `tenant_id`, `source_event_id`, `source_global_position`, `topic`, `message_key`, `payload`, `metadata`, and `status = 'pending'`; generate `outbox_id` in Rust with `Uuid::now_v7()`. Use `ON CONFLICT (tenant_id, source_event_id, topic) DO NOTHING` so late duplicate effects remain idempotent, but do not suppress invalid FK or malformed source-event validation. Do not call any publisher from `sql.rs`.

    Extend `crates/es-store-postgres/tests/outbox.rs` with `append_creates_outbox_rows_atomically`, `append_duplicate_replay_does_not_duplicate_outbox_rows`, and `append_conflict_rolls_back_outbox_rows`. Use `AppendRequest::new_with_outbox` and query `outbox_messages` directly to assert row count, `source_global_position`, `topic`, `message_key`, `tenant_id`, and `status = 'pending'`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| append request -> database transaction | Caller-supplied outbox messages are persisted with source events. |
| command dedupe -> outbox insert | Duplicate command replay must not create new external effects. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | append outbox rows | mitigate | Use `request.command_metadata.tenant_id` for every inserted row; never accept a separate outbox tenant. |
| T-06-02 | Tampering | duplicate external effects | mitigate | Validate `source_event_id` against the current append events and use `ON CONFLICT (tenant_id, source_event_id, topic) DO NOTHING`. |
| T-06-03 | Denial of Service | poison retry loop | accept | Retry loop is not executed in append; Plan 04 owns bounded retry processing. |
| T-06-04 | Tampering | SQL injection | mitigate | Use static SQLx queries and `.bind(...)` only; acceptance criteria scans `sql.rs` for forbidden publisher calls. |
| T-06-05 | Tampering | workflow replay duplicate commands | mitigate | Atomic source outbox rows preserve one durable workflow trigger per source event/topic before Plan 05 consumes committed events. |
</threat_model>
