<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| outbox contracts -> PostgreSQL | Validated DTOs become durable tenant-owned rows. |
| dispatcher workers -> row claims | Multiple worker processes compete for due pending rows. |
| process managers -> offset table | Workflow progress is persisted per tenant and process-manager name. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | `outbox_messages` queries | mitigate | Include `tenant_id` in table columns, unique keys, indexes, and every repository predicate. Test `outbox_repository_filters_by_tenant`. |
| T-06-02 | Tampering | Duplicate external effects | mitigate | Enforce `UNIQUE (tenant_id, source_event_id, topic)` in the migration and verify duplicate inserts in `outbox_is_idempotent_by_source_event_and_topic`. |
| T-06-03 | Denial of Service | Poison retry loop | mitigate | Persist `attempts`, `available_at`, `last_error`, and `failed` status; `schedule_retry` must call `mark_failed` when attempts reach `RetryPolicy::max_attempts()`. |
| T-06-04 | Tampering | SQL injection | mitigate | All repository SQL uses static strings plus `.bind(...)`; no `format!` or string interpolation for SQL values. |
| T-06-05 | Tampering | Workflow replay duplicate commands | mitigate | Persist tenant-scoped `process_manager_offsets`; Plan 05 will combine offsets with deterministic command idempotency keys. |
</threat_model>
