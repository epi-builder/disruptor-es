<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| committed events -> workflow commands | Durable event payloads become follow-up command envelopes. |
| process manager -> command gateway | Workflow code crosses into runtime admission and waits for commit-gated replies. |
| process manager -> offsets | Durable offset advancement records workflow completion. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | follow-up command metadata | mitigate | Follow-up `CommandMetadata.tenant_id` is copied from `ProcessEvent.tenant_id`; no workflow command can supply a different tenant. |
| T-06-02 | Tampering | duplicate external effects | accept | This plan issues commands, not publisher effects; outbox dispatch idempotency remains in Plans 01 and 04. |
| T-06-03 | Denial of Service | poison workflow event | mitigate | Decode failures return `OutboxError::PayloadDecode` and do not advance offsets; command overload/unavailable returns `CommandSubmit` and stops the batch. |
| T-06-04 | Tampering | SQL injection | accept | This plan has no SQL; offset persistence is through `ProcessManagerOffsetStore`. |
| T-06-05 | Tampering | workflow replay duplicate commands | mitigate | Idempotency keys are deterministic from process-manager name, source event ID, target aggregate, and action; offsets advance only after command replies finish. |
</threat_model>
