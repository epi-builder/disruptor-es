  <behavior>
    - Empty claims return `DispatchOutcome::Idle`.
    - Successful publish calls `mark_published` after `Publisher::publish` returns `Ok(())`.
    - Publisher failure calls `schedule_retry`, does not call `mark_published`, and increments either `retried` or `failed` from `RetryScheduleOutcome`.
    - Published envelopes preserve `OutboxMessage::idempotency_key()` and `message_key`.
  </behavior>
  <action>
    Add `mod dispatcher;` and public re-exports for `dispatch_once` and `OutboxStore` to `crates/es-outbox/src/lib.rs`.

    Create `crates/es-outbox/src/dispatcher.rs` with a storage-neutral `pub trait OutboxStore: Clone + Send + Sync + 'static` using `futures::future::BoxFuture` methods:
    `claim_pending(&self, tenant_id: es_core::TenantId, worker_id: WorkerId, limit: DispatchBatchLimit) -> BoxFuture<'_, OutboxResult<Vec<OutboxMessage>>>`,
    `mark_published(&self, tenant_id: es_core::TenantId, outbox_id: uuid::Uuid) -> BoxFuture<'_, OutboxResult<()>>`,
    and `schedule_retry(&self, tenant_id: es_core::TenantId, outbox_id: uuid::Uuid, error: String, retry_policy: RetryPolicy) -> BoxFuture<'_, OutboxResult<RetryScheduleOutcome>>`.
    Implement `pub async fn dispatch_once<S, P>(store: &S, publisher: &P, tenant_id: TenantId, worker_id: WorkerId, limit: DispatchBatchLimit, retry_policy: RetryPolicy) -> OutboxResult<DispatchOutcome> where S: OutboxStore, P: Publisher`. The loop must publish `message.publish_envelope()`, call `mark_published` only on `Ok(())`, call `schedule_retry` on `Err(error)`, count `published`, increment `retried` when storage returns `RetryScheduleOutcome::RetryScheduled`, increment `failed` when storage returns `RetryScheduleOutcome::Failed`, and return `DispatchOutcome::Published { published }` when all claimed rows publish successfully or `DispatchOutcome::Partial { published, retried, failed }` when any retry or failed transition occurs. Do not use a constant zero failed count; PostgreSQL storage can move a row to failed status at max attempts.

    Add dispatcher unit tests in `dispatcher.rs` using fake stores and `InMemoryPublisher` for `dispatcher_returns_idle_when_no_rows_are_claimed`, `dispatcher_marks_success_after_publish`, `dispatcher_schedules_retry_after_publish_failure`, `dispatcher_reports_failed_when_retry_policy_is_exhausted`, and `dispatcher_preserves_idempotency_key`.

    Implement `es_outbox::OutboxStore` for `PostgresOutboxStore` in `crates/es-store-postgres/src/outbox.rs`. Use a fixed lock duration of 30 seconds when delegating to the existing repository `claim_pending`. Map `StoreError` into `OutboxError::Store { message }`.

    Extend `crates/es-store-postgres/tests/outbox.rs` with `dispatcher_marks_successful_rows_published`, `dispatcher_schedules_failed_publish_for_retry`, and `dispatcher_reports_failed_rows_at_max_attempts`. Use `InMemoryPublisher` for the successful path and `push_failure("broker down")` for the retry/failed paths.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PostgreSQL outbox -> dispatcher | Durable pending rows become publish attempts. |
| dispatcher -> publisher | External effects can occur and may be retried after crashes or failures. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | dispatcher tenant scope | mitigate | `dispatch_once` accepts one `TenantId`; `OutboxStore` calls receive that tenant and PostgreSQL implementation filters by tenant. |
| T-06-02 | Tampering | duplicate external effects | mitigate | Publisher receives deterministic `idempotency_key`; unit tests verify the key is preserved and `InMemoryPublisher` dedupes repeated keys. |
| T-06-03 | Denial of Service | poison retry loop | mitigate | Dispatcher passes `RetryPolicy` to `schedule_retry`; PostgreSQL implementation transitions rows to `failed` at max attempts. |
| T-06-04 | Tampering | SQL injection | mitigate | Dispatcher has no SQL; PostgreSQL implementation from Plan 02 uses bound SQLx parameters. |
| T-06-05 | Tampering | workflow replay duplicate commands | accept | Dispatcher does not issue workflow commands; Plan 05 owns deterministic process-manager command idempotency. |
</threat_model>
