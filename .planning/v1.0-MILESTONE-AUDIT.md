## Blockers

### Runtime Idempotency Is Not End-To-End

`DedupeCache::get` exists, and committed append results are recorded after a successful append, but `ShardState::process_next_handoff` does not check the dedupe cache before rehydrating state and calling `A::decide`.

That means a repeated command with the same tenant/idempotency key can be evaluated against already-mutated aggregate state and fail domain validation before reaching `PostgresEventStore::append`, where durable dedupe replay would otherwise return the original committed result.

Affected requirements: STORE-03, RUNTIME-03, RUNTIME-05, INT-04, API-01, API-03.

Evidence:

| File | Evidence |
|---|---|
| `crates/es-runtime/src/shard.rs:171` | Loads current state from cache/rehydration. |
| `crates/es-runtime/src/shard.rs:188` | Calls aggregate `decide` before any dedupe lookup. |
| `crates/es-runtime/src/shard.rs:245` | Reaches store append only after decision and event encoding. |
| `crates/es-runtime/src/cache.rs:82` | `DedupeCache::get` exists but is not used in the processing path. |

Broken flow: duplicate command retry through API/runtime/store.

At-risk flow: process-manager crash/retry after follow-up commands but before offset advancement.
