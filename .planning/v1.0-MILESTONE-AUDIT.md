## Blockers

### B1: Shard Aggregate Cache Is Not Tenant-Scoped

`AggregateCache` stores aggregate state by `StreamId` only. Event-store reads and dedupe are tenant-scoped, but `ShardState::process_next_handoff` can use a cached state before tenant-specific rehydration. Two tenants using the same stream id and landing on the same shard can therefore share hot aggregate state.

Affected requirements: RUNTIME-03, RUNTIME-05, RUNTIME-06, STORE-04, DOM-05.

| File | Evidence |
|---|---|
| `crates/es-runtime/src/cache.rs:7` | `AggregateCache` is a `HashMap<StreamId, A::State>`. |
| `crates/es-runtime/src/shard.rs:218` | Processing reads aggregate state through the cache path. |
| `crates/es-runtime/src/router.rs:45` | Routing is tenant-aware, but shard-local state cache still needs tenant identity in its key. |

### B2: Process-Manager Idempotency Keys Collide for Duplicate Product Lines

The commerce process manager builds reserve/release idempotency keys from process-manager name, source event id, action, and product id. If an order contains two lines for the same product, the second follow-up command can reuse the first line's key and replay the wrong prior reserve/release result.

Affected requirements: INT-04, STORE-03, RUNTIME-05, DOM-04, DOM-05.

| File | Evidence |
|---|---|
| `crates/app/src/commerce_process_manager.rs:68` | The reserve loop iterates order lines without a stable line ordinal. |
| `crates/app/src/commerce_process_manager.rs:78` | Reserve key includes source event and product id only. |
| `crates/app/src/commerce_process_manager.rs:115` | Release key uses the same source event plus product id shape. |
